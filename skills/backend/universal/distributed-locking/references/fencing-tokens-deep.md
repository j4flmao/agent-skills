# Fencing Token Deep Dive

## The Problem Fencing Tokens Solve

Without fencing tokens, a lock that has timed out can still have a stale holder that continues writing to shared storage, corrupting data.

```
Stale Instance A: [holds lock] [GC pause: 8s]
                     ↑ lock expires here
Instance B:                     [acquires lock] [writes with token 2]
Instance A:                                          [resumes] [writes with token 1] ← CORRUPTION
```

## How Fencing Tokens Work

A fencing token is a monotonically increasing integer generated by the lock service when a lock is acquired. The token is presented with every write to shared storage, which rejects writes with stale (non-increasing) tokens.

## Implementation Patterns

### Database-Level Fencing (PostgreSQL)
```typescript
class FencedWriteService {
  async writeWithFencing(
    resourceId: string,
    data: object,
    operation: (conn: DbClient, fencingToken: number) => Promise<void>,
  ): Promise<void> {
    const lock = await this.lockService.acquire(resourceId, 5000);
    if (!lock) throw new Error('Could not acquire lock');

    try {
      // Read current fencing token
      const { rows: [row] } = await this.db.query(
        'SELECT fencing_token FROM resources WHERE id = $1 FOR UPDATE',
        [resourceId]
      );
      const currentToken = row?.fencing_token ?? 0;
      const newToken = Math.max(currentToken, lock.fencingToken);

      // Execute operation with fencing check
      await this.db.query(
        'UPDATE resources SET data = $1, fencing_token = $2, updated_at = NOW() WHERE id = $3 AND fencing_token < $2',
        [JSON.stringify(data), newToken, resourceId]
      );

      // Verify write was accepted
      const { rowCount } = await this.db.query(
        'SELECT 1 FROM resources WHERE id = $1 AND fencing_token = $2',
        [resourceId, newToken]
      );
      if (rowCount === 0) throw new Error('Fencing token rejected — stale lock');
    } finally {
      await lock.release();
    }
  }
}
```

### Monotonic Fencing Token Generator
```typescript
interface FencingTokenGenerator {
  nextToken(): Promise<number>;
}

class RedisFencingTokenGenerator implements FencingTokenGenerator {
  constructor(private redis: RedisClient) {}

  async nextToken(): Promise<number> {
    return this.redis.incr('global:fencing:counter');
  }
}

class PostgresFencingTokenGenerator implements FencingTokenGenerator {
  constructor(private pool: Pool) {}

  async nextToken(): Promise<number> {
    const { rows: [row] } = await this.pool.query(
      "SELECT nextval('fencing_token_seq') AS token"
    );
    return row.token;
  }
}
```

### ZooKeeper Sequential Fencing
ZooKeeper's sequential nodes provide fencing tokens natively:

```typescript
class ZooKeeperFencedLock {
  async acquire(resourcePath: string): Promise<FencedLock> {
    // Create an ephemeral sequential node
    // ZooKeeper assigns a monotonically increasing sequence number
    const createdPath = await this.zk.create(
      `/locks/${resourcePath}/lock-`,
      Buffer.from(process.env.HOSTNAME),
      CreateMode.EPHEMERAL_SEQUENTIAL,
    );

    // Extract the sequence number as the fencing token
    const sequence = parseInt(createdPath.split('-').pop()!, 10);

    // Check if we're the lowest sequence — if so, we hold the lock
    const children = await this.zk.getChildren(`/locks/${resourcePath}`);
    const lowestSequence = Math.min(
      ...children.map(c => parseInt(c.split('-').pop()!, 10))
    );

    if (sequence === lowestSequence) {
      return new FencedLock(resourcePath, sequence, this);
    }

    // Wait for the previous lock holder to release
    await this.watchPreviousNode(resourcePath, sequence);
    return new FencedLock(resourcePath, sequence, this);
  }
}
```

## When Fencing Tokens Are Required

| Scenario | Fencing Required? | Alternative |
|---|---|---|
| Protecting a DB write without version column | YES | — |
| Protecting a DB write with optimistic locking | Optional | Version column serves as fencing |
| File system writes | YES | — |
| Idempotent API calls | No | Idempotency key suffices |
| Read-only operations | No | No state change |
| Appending to a log | No | Append is idempotent |

## Testing Fencing Token Safety

### Chaos Engineering Test
```typescript
describe('Fencing Token Safety', () => {
  it('prevents stale lock holder from writing', async () => {
    // Step 1: Instance A acquires lock with token 1
    const lockA = await lockService.acquire('resource-1', 100);
    await lockService.setFencingToken('resource-1', 1);
    
    // Step 2: Lock expires, Instance B acquires token 2
    await sleep(200);
    const lockB = await lockService.acquire('resource-1', 5000);
    await lockService.setFencingToken('resource-1', 2);
    
    // Step 3: Instance B writes successfully
    const bResult = await writeWithFencing('resource-1', { data: 'B' }, 2);
    expect(bResult).toBe(true);
    
    // Step 4: Stale Instance A tries to write with token 1
    const aResult = await writeWithFencing('resource-1', { data: 'A' }, 1);
    expect(aResult).toBe(false); // Must be rejected!
    
    // Verify final state is from Instance B
    const state = await readResource('resource-1');
    expect(state.data).toBe('B');
  });
});
```

## Fencing Token Best Practices

1. **Always monotonically increasing**: Tokens must never decrease. Use a database sequence, Redis INCR, or ZooKeeper sequential nodes.
2. **Per-resource or global**: Per-resource tokens are more efficient (less contention). Global tokens are simpler.
3. **Include in all writes**: Every write to shared storage must include the fencing token check.
4. **Reject on mismatch**: If the token is not strictly greater than the last seen token, reject the write.
5. **CRDTs as alternative**: Conflict-free Replicated Data Types (CRDTs) can eliminate the need for fencing in some cases.
