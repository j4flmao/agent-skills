# Prompt Performance Optimization

## 1. The Performance Trilemma in Prompt Engineering

Optimizing few-shot prompts involves balancing three competing factors:
1.  **Accuracy (Quality)**: Does the model generate the correct, desired output?
2.  **Latency (Speed)**: How long does the user have to wait for the response?
3.  **Cost (Efficiency)**: How many tokens are consumed, and at what price?

Improving one factor often degrades another. Adding more few-shot examples increases Accuracy but hurts Latency and Cost. Using a smaller model improves Latency and Cost but may reduce Accuracy.

## 2. Key Performance Indicators (KPIs)

To optimize, we must first measure.

### 2.1 Latency Metrics
-   **Time to First Token (TTFT)**: Crucial for streaming applications. Measures the responsiveness of the system.
-   **Time per Output Token (TPOT)**: Measures the generation speed. Varies heavily by model provider and current server load.
-   **Total Request Latency**: The end-to-end time, including network transit and prompt processing.

### 2.2 Cost Metrics
-   **Input Token Volume**: The total number of tokens in the prompt (system + examples + query).
-   **Output Token Volume**: The tokens generated by the model.
-   **Cost per Request**: Calculated based on provider pricing (e.g., $0.01 / 1K input tokens).

### 2.3 Accuracy Metrics
-   **Exact Match (EM)**: Did the output exactly match the expected ground truth? (Useful for classification/extraction).
-   **Semantic Similarity**: How close is the meaning of the output to the ground truth? (Measured via embeddings).
-   **Format Adherence Rate**: What percentage of responses successfully parse into the required JSON/YAML structure?

## 3. Optimization Strategy 1: Prompt Caching

Many modern LLM providers (e.g., Anthropic, OpenAI) support prompt caching. This is the single most effective way to optimize few-shot prompts.

### 3.1 How it Works
When you send a large prompt (like a complex system instruction + 20 few-shot examples), the provider caches the processed attention states of the initial tokens. If subsequent requests start with the exact same sequence of tokens, the provider can skip processing them, dramatically reducing Latency and Cost.

### 3.2 Implementation Rules for Caching
1.  **Static Prefixing**: Structure your prompt so the static parts (System Instruction + Few-Shot Examples) are at the absolute beginning of the prompt.
2.  **Dynamic Suffixing**: Place dynamic parts (User Query, current context) at the very end.
3.  **Order Matters**: If you dynamically select few-shot examples (as per `example-selection-architectures.md`), caching is defeated because the prefix changes every time. You must balance dynamic selection against caching benefits.

```text
[---- CACHEABLE PREFIX ----] [---- DYNAMIC SUFFIX ----]
System Prompt + 10 Examples  + User Query
```

## 4. Optimization Strategy 2: Example Distillation

Do you really need 10 full-length examples? Example Distillation involves compressing the examples to extract maximum signal with minimal tokens.

### 4.1 Techniques
-   **Remove Boilerplate**: Strip out polite filler words from the examples.
-   **Data Serialization**: Instead of verbose text descriptions, represent examples as compact JSON or CSV.
-   **Negative Examples**: Sometimes, providing one clear negative example ("DO NOT do this: X") is more effective and uses fewer tokens than three positive examples.

### 4.2 Distillation Example

**Verbose (Bad):**
`User: I would like to know the capital of France.`
`Assistant: That is a great question. The capital of France is Paris.`

**Distilled (Good):**
`Q: Capital of France?`
`A: Paris`

## 5. Advanced Implementation (TypeScript): Metrics Interceptor

To track performance, implement an interceptor or wrapper around your LLM calls.

```typescript
import { performance } from 'perf_hooks';

export interface LLMRequest {
  prompt: string;
  model: string;
}

export interface LLMResponse {
  text: string;
  usage: { inputTokens: number; outputTokens: number };
}

export interface PerformanceMetrics {
  totalLatencyMs: number;
  inputTokens: number;
  outputTokens: number;
  estimatedCostCent: number;
  model: string;
}

export class LLMClientWithMetrics {
  constructor(private readonly baseClient: any) {}

  private calculateCost(model: string, input: number, output: number): number {
    // Pricing as of early 2024 (Example)
    const pricing = {
      'gpt-4-turbo': { input: 1.0, output: 3.0 }, // Cents per 1k
      'gpt-3.5-turbo': { input: 0.05, output: 0.15 }
    };
    const rate = pricing[model as keyof typeof pricing] || { input: 0, output: 0 };
    return (input / 1000) * rate.input + (output / 1000) * rate.output;
  }

  public async execute(request: LLMRequest): Promise<{response: LLMResponse, metrics: PerformanceMetrics}> {
    const startTime = performance.now();
    
    // Execute actual LLM call
    const response: LLMResponse = await this.baseClient.generate(request);
    
    const endTime = performance.now();
    const totalLatencyMs = endTime - startTime;

    const metrics: PerformanceMetrics = {
      totalLatencyMs,
      inputTokens: response.usage.inputTokens,
      outputTokens: response.usage.outputTokens,
      model: request.model,
      estimatedCostCent: this.calculateCost(
        request.model, 
        response.usage.inputTokens, 
        response.usage.outputTokens
      )
    };

    // Log metrics asynchronously to avoid blocking
    this.logMetrics(metrics);

    return { response, metrics };
  }

  private logMetrics(metrics: PerformanceMetrics) {
    // Send to Datadog, Prometheus, etc.
    console.log(`[METRICS] Latency: ${metrics.totalLatencyMs.toFixed(0)}ms | Cost: ¢${metrics.estimatedCostCent.toFixed(4)}`);
  }
}
```

## 6. Optimization Strategy 3: Output Formatting Optimization

The format you request the model to output heavily impacts Latency and Cost. Generating tokens takes time.

-   **Avoid XML/HTML**: These formats are token-heavy due to closing tags.
-   **Prefer JSON over YAML**: While YAML is readable, models are often more highly trained on JSON, leading to fewer syntax errors and sometimes faster generation.
-   **Use Short Keys**: In JSON, request short keys. `{"t": "value"}` uses fewer tokens than `{"target_extraction_value": "value"}`.

### 6.1 JSON Schema Optimization

When using function calling or JSON mode, the schema itself consumes input tokens.

**Inefficient Schema:**
```json
{
  "properties": {
    "user_first_name": { "type": "string", "description": "The first name of the user mentioned in the text" },
    "user_last_name": { "type": "string", "description": "The last name of the user mentioned in the text" }
  }
}
```

**Optimized Schema:**
```json
{
  "properties": {
    "name": { "type": "string", "description": "User's full name" }
  }
}
```

## 7. Model Routing (Dynamic Fallback)

Not every query requires a massive, expensive model. Model routing optimizes cost and latency by using the smallest capable model for the task.

1.  **Complexity Classifier**: Use a fast, cheap model (or rule-based system) to classify the difficulty of the query.
2.  **Routing Logic**: 
    - Easy queries -> GPT-3.5 or Claude Haiku (Low Latency, Low Cost).
    - Hard queries -> GPT-4 or Claude Opus (High Accuracy).
3.  **Few-Shot Adaptation**: The few-shot examples must be tailored to the target model. Smaller models often need *more* examples to achieve the same accuracy as larger models.

## 8. Conclusion

Performance optimization in prompt engineering is a continuous cycle of measurement and refinement. By implementing strict metric tracking, leveraging prompt caching, distilling examples, and dynamically routing requests, engineering teams can build LLM applications that are not only highly accurate but also fast and economically viable at scale.
