# Svelte Testing Strategies

## Purpose
This document serves as a comprehensive, deep-dive reference guide for Svelte Testing Strategies within the Svelte ecosystem.
Focus areas include: Vitest, Playwright, Component Testing vs E2E, mocking.

## Core Principles
1. **Reactivity First**: Leverage Svelte's compiler to handle reactivity efficiently.
2. **Minimal Overhead**: Keep the DOM updates surgical and the JS payload small.
3. **Progressive Enhancement**: Ensure baseline functionality works without JavaScript where applicable.
4. **Predictable State Flow**: Maintain clear ownership of state, favoring top-down data flow.
5. **Component Encapsulation**: Keep styles, markup, and logic scoped unless globally required.

## Svelte 5 Runes Paradigm
With the introduction of Svelte 5, the paradigm shifts from label-based reactivity (`$:`) and store subscriptions (`$store`) to Runes (`$state`, `$derived`, `$effect`).
This fundamentally alters how we approach Svelte Testing Strategies.

### The Shift from Svelte 4
In Svelte 4, reactivity was bound to the component instance. Variables declared with `let` at the top level of a component were reactive *within that component*.
In Svelte 5, Runes decouple reactivity from the component. You can use `$state` in vanilla `.js` or `.ts` files, enabling incredibly powerful, universal reactivity patterns.

## Detailed Architectural Overview
```text
TESTING PYRAMID
===============

         / \
        /   \      E2E TESTS (Playwright)
       /     \     - Full user journeys
      /-------\    - Real browser
     /         \
    /   INTEG.  \  COMPONENT TESTS (Vitest + Testing Library)
   /    TESTS    \ - Interaction & DOM updates
  /---------------\
 /                 \
/    UNIT TESTS     \ CORE LOGIC (Vitest)
--------------------- - Helper functions, stores, derived state
```

## Code Examples & Real-world Usage
```svelte
// Vitest Component Test
import { render, screen, fireEvent } from '@testing-library/svelte';
import Counter from './Counter.svelte';
import { describe, it, expect } from 'vitest';

describe('Counter Component', () => {
    it('renders with initial state', () => {
        render(Counter, { initialCount: 0 });
        expect(screen.getByText('Count: 0')).toBeInTheDocument();
    });

    it('increments when button is clicked', async () => {
        render(Counter, { initialCount: 0 });
        const button = screen.getByRole('button', { name: /increment/i });
        
        await fireEvent.click(button);
        
        expect(screen.getByText('Count: 1')).toBeInTheDocument();
    });
});
```

## Advanced Patterns & Techniques
### 1. Granular Updates
By using `$derived` correctly, you prevent unnecessary re-computations.
### 2. Effect Lifecycle Management
Always return a cleanup function from `$effect` if you subscribe to external events or instantiate timers.
### 3. Server-side Integration
When using SvelteKit, remember that `$state` and other runes are evaluated on the server during SSR. Avoid browser-specific APIs (like `window` or `document`) in the root of your script blocks unless wrapped in `$effect` or `onMount`.

## Extended Troubleshooting Guide
| Symptom | Primary Cause | Mitigation Action |
|---------|---------------|-------------------|
| State not updating | Mutating an object property without reassignment (Svelte 4) or missing `$state` rune (Svelte 5) | Use `$state` for deep reactivity or reassign the object in Svelte 4. |
| Memory leaks | Missing cleanup in `$effect` or `onMount` | Return a cleanup function that removes listeners/clears intervals. |
| Hydration mismatch | SSR HTML differs from Client HTML (often due to date formatting or browser-only APIs) | Move browser-dependent rendering into an `$effect` or check `browser` from `$app/environment`. |
| Unnecessary re-renders | Derived state causing cascading updates | Memoize complex calculations or ensure `$derived` dependencies are stable. |
| Infinite loops | `$effect` updating a state that triggers the same `$effect` | Untrack the state update using `untrack()` or rethink the reactivity flow. |
| Store subscription errors | Using `$` prefix outside `.svelte` files | Use `get(store)` or refactor to Svelte 5 universal Runes (`$state`). |

## Workflow Steps for Implementation
1. **Analyze Requirements**: Determine the scope and scale of the necessary implementation.
2. **Design the Architecture**: Sketch the component tree and data flow.
3. **State Modeling**: Define `$state` and `$derived` variables.
4. **Component Implementation**: Write the markup and logic, utilizing the modeled state.
5. **Integration**: Connect with APIs or server-side loaders (+page.server.ts).
6. **Testing & Optimization**: Write Vitest/Playwright tests and profile performance.

## Rules and Guidelines
- NEVER mutate state directly outside of the Svelte reactivity system.
- ALWAYS prefer `$derived` over manually updating state in an `$effect`.
- DO NOT overuse `$effect`; most logic should be derived or event-driven.
- KEEP components small and focused on a single responsibility.
- USE TypeScript for enhanced developer experience and type safety.
