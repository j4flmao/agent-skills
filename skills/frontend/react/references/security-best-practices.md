## 1. Cross-Site Scripting (XSS) Prevention

React inherently protects against XSS by automatically escaping string variables rendered in JSX. However, certain APIs bypass this protection.

### `dangerouslySetInnerHTML` Risks
Using `dangerouslySetInnerHTML` injects raw HTML into the DOM, creating a massive XSS vector if the input is untrusted.

```tsx
// DANGEROUS: Untrusted input executed as script
const untrustedInput = "<img src='x' onerror='alert(1)'>";
return <div dangerouslySetInnerHTML={{ __html: untrustedInput }} />;
```

### Mitigation: DOMPurify
Always sanitize HTML content before injecting it.

```tsx
import DOMPurify from 'dompurify';

function SafeHtmlRenderer({ htmlContent }: { htmlContent: string }) {
  // Sanitize the HTML string
  const cleanHtml = DOMPurify.sanitize(htmlContent);

  return <div dangerouslySetInnerHTML={{ __html: cleanHtml }} />;
}
```

### URL Injection
Ensure URLs used in `href` or `src` attributes use safe protocols (`http://`, `https://`, `mailto:`). Block `javascript:` protocols.

```tsx
// DANGEROUS
const userLink = "javascript:alert('XSS')";
<a href={userLink}>Click Me</a>

// SAFE Validation
function isSafeUrl(url: string) {
  return url.startsWith('http://') || url.startsWith('https://');
}
```

## 2. Cross-Site Request Forgery (CSRF) in SSR

CSRF tricks a user's browser into executing unwanted actions on a trusted site. It is primarily a concern when using cookie-based authentication.

### Mitigation in Next.js / Server Environments
1. **SameSite Cookie Attribute:** Set cookies with `SameSite=Lax` or `SameSite=Strict`.
2. **CSRF Tokens:** For state-changing operations (POST, PUT, DELETE), require a CSRF token generated by the server and included in the request headers.

```javascript
// Next.js API Route example setting a secure cookie
export default function handler(req, res) {
  res.setHeader('Set-Cookie', 'session_id=123; HttpOnly; Secure; SameSite=Strict; Path=/');
  res.status(200).json({ message: 'Success' });
}
```

## 3. Authentication: JWT vs Session Cookies

### JWT (JSON Web Tokens)
- **Pros:** Stateless, easily used across multiple microservices.
- **Cons:** Hard to invalidate before expiration, payload is readable by anyone (not encrypted, only signed).
- **Storage:** Storing JWTs in `localStorage` exposes them to XSS attacks. Store them in `HttpOnly` cookies.

### Session Cookies
- **Pros:** Stored on the server, easily revocable, inherently more secure against XSS if marked `HttpOnly`.
- **Cons:** Requires database lookup for session validation.

### Recommended Approach (Next.js / NextAuth)
Use `HttpOnly`, `Secure`, `SameSite` cookies to store sessions. Do not expose tokens to the client-side JavaScript.

```typescript
// NextAuth.js configuration pattern
export const authOptions = {
  providers: [ /* ... */ ],
  session: {
    strategy: "jwt", // Uses JWE (encrypted) in a secure cookie
  },
  callbacks: {
    async session({ session, token }) {
      // Expose only necessary non-sensitive info to the client
      session.user.id = token.sub;
      return session;
    }
  }
}
```

## 4. Content Security Policy (CSP) Headers

A CSP restricts the resources (scripts, images, styles) that the browser is allowed to load for a given page.

### Next.js Implementation via `next.config.js`

```javascript
// next.config.js
const ContentSecurityPolicy = `
  default-src 'self';
  script-src 'self' 'unsafe-eval' 'unsafe-inline' https://apis.google.com;
  style-src 'self' 'unsafe-inline';
  img-src * blob: data:;
  media-src 'none';
  connect-src 'self' https://api.mybackend.com;
  font-src 'self';
`;

const securityHeaders = [
  {
    key: 'Content-Security-Policy',
    value: ContentSecurityPolicy.replace(/\s{2,}/g, ' ').trim()
  },
  { key: 'X-Frame-Options', value: 'DENY' },
  { key: 'X-Content-Type-Options', value: 'nosniff' },
  { key: 'Referrer-Policy', value: 'strict-origin-when-cross-origin' }
];

module.exports = {
  async headers() {
    return [
      {
        source: '/(.*)',
        headers: securityHeaders,
      },
    ];
  },
};
```

## 5. Dependency Vulnerability Scanning

Third-party dependencies represent a significant attack surface (Supply Chain Attacks).

### Practices:
1. **Regular Audits:** Run `npm audit` or `yarn audit` frequently in CI/CD pipelines.
2. **Automated Tools:** Use Dependabot or Snyk to automatically create PRs for vulnerable packages.
3. **Lockfiles:** Always commit `package-lock.json` or `yarn.lock` to ensure deterministic builds.

```yaml
# GitHub Actions example for running audit
name: Security Audit
on: [push, pull_request]
jobs:
  audit:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Run npm audit
        run: npm audit --audit-level=high
```

## 6. Secure Data Handling

- **Never bundle secrets:** Environment variables starting with `NEXT_PUBLIC_`, `REACT_APP_`, or `VITE_` are exposed to the browser. Never prefix database passwords, API keys, or JWT secrets with these prefixes.
- **Clear Sensitive State:** If using global state (Redux/Zustand), clear sensitive data upon user logout to prevent leakage if the application is left open.
