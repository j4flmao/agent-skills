# Vector Databases Fundamentals

## What Is a Vector Database

A vector database stores, indexes, and searches high-dimensional vector embeddings — numerical representations of unstructured data (text, images, audio) produced by embedding models. Unlike traditional databases that search by exact keyword match, vector databases retrieve semantically similar items using distance metrics in embedding space.

### Core Capabilities
- **ANN (Approximate Nearest Neighbor) search** — fast similarity search at scale
- **Metadata filtering** — combine vector similarity with structured filters
- **CRUD operations** — insert, update, delete individual vectors
- **Index building** — organize vectors for sub-linear search time
- **Distributed storage** — shard/replicate across nodes for scale and HA

## How Vector Databases Work

### Embedding Pipeline
```
Raw Data → Embedding Model → Vector (float[]) + Metadata (JSON) → Vector DB → Index
```

Each vector is a point in high-dimensional space (typically 384–3072 dimensions). Semantically similar items cluster together. The database builds an index structure that enables fast nearest-neighbor lookup.

### Search Flow
1. Query embedding generated by the same model used for ingestion
2. Distance computed between query and indexed vectors (using configured metric)
3. Top-K most similar vectors returned, optionally filtered by metadata
4. Results ranked by similarity score

## Vector Dimensions and Layout

| Dimension | Typical Use | Memory per Vector (float32) |
|-----------|-------------|----------------------------|
| 384 | Lightweight semantic search, all-MiniLM | 1.5 KB |
| 768 | Sentence-BERT, ada-002 | 3 KB |
| 1024 | BGE-large, Cohere embed-v3 | 4 KB |
| 1536 | OpenAI text-embedding-3-large | 6 KB |
| 3072 | Cohere embed-english-v3.0 | 12 KB |

Memory = `dim × 4 bytes × N vectors` for raw storage. Index adds 20-200% overhead depending on index type.

## Distance Metrics

### Cosine Similarity
```
d = 1 - (A · B) / (|A| × |B|)
```
Range: [0, 2]. 0 = identical direction, 2 = opposite.
**Default for text embeddings** (OpenAI, Sentence-Transformers, BGE).
Vectors are normalized to unit length during training — cosine on normalized vectors is equivalent to inner product.

### Euclidean (L2)
```
d = sqrt(Σ(A_i - B_i)²)
```
Range: [0, ∞). 0 = identical.
Use when: embedding model outputs unnormalized vectors, spatial/coordinate data.
Sensitive to scale — normalize if arbitrary.

### Dot Product (IP)
```
d = A · B = Σ A_i × B_i
```
Range: (-∞, ∞). Higher = more similar.
Use when: model trained with dot product loss (Cohere, recommendation models, some multimodal).
On unit vectors, equivalent to cosine.

### Selection Rule
**Always match the distance metric to the embedding model's training loss function.**
- OpenAI embeddings → cosine
- Cohere embeddings → dot product
- Sentence Transformers → cosine
- BGE embeddings → cosine
- Custom models → check training loss (cosine, triplet, contrastive, dot product)

## ANN vs Exact Search

| | Exact (Flat/Brute-force) | ANN (HNSW, IVF, PQ) |
|---|---|---|
| Recall | 100% | 90-99.9% configurable |
| Speed (1M vectors) | O(N) ~50ms | O(log N) ~2-10ms |
| Memory | Full vectors | Full vectors + index overhead |
| Use case | <10K vectors, ground-truth eval | >10K vectors, production search |

**ANN is always an approximation** — you trade a small recall loss for orders-of-magnitude latency improvement. Tune index parameters (efSearch, nprobe) to dial in the recall-latency sweet spot.

## Index Types Overview

### Flat (Brute Force)
- No index structure; computes distance against every vector
- 100% recall guaranteed
- O(N) time, O(N) memory
- Use only for: small datasets (<10K), exact evaluation, prototyping

### HNSW (Hierarchical Navigable Small World)
- Multi-layer graph: sparse upper layers (long-range edges), dense lower layers (short-range)
- O(log N) search time
- High memory due to graph adjacency storage (~1.5× vector size)
- Best speed/recall trade-off for <10M vectors
- Parameters: M (connections/node), efConstruction (build quality), efSearch (query scope)

### IVF (Inverted File Index)
- K-means to partition space into Voronoi cells; search nearest N clusters
- Lower memory than HNSW (no graph overhead)
- O(sqrt(N)) effective search time
- Best for >10M vectors with memory constraints
- Parameters: nlist (cluster count), nprobe (cells to visit)

### IVF + PQ (Product Quantization)
- Compress stored vectors via sub-vector quantization (8:1 to 16:1 compression)
- Dramatically reduces memory at cost of recall (0.85-0.93)
- Parameters: M sub-vectors, nbits per codebook entry
- Use when: memory budget is tight, recall >0.9 acceptable

### DiskANN
- SSD-backed Vamana graph; memory-maps index from disk
- Billion-scale with <2 GB RAM
- O(log N) with higher latency (disk reads)
- Parameters: beamwidth (build), search_list_size (query)
- Use when: dataset >100M vectors, doesn't fit in RAM

## Core CRUD Operations

### Insert
```python
collection.upsert(vectors=[
    ("id1", [0.1, 0.2, ...], {"category": "docs", "source": "file1.pdf"}),
    ("id2", [0.3, 0.4, ...], {"category": "code", "source": "src/main.py"}),
])
```
- Most databases support batch upsert (idempotent: insert or update)
- Bulk insert in batches of 100-1000 for throughput
- Index updates happen inline or asynchronously depending on DB

### Read (Fetch by ID)
```python
result = collection.fetch("id1")
# Returns: {"id": "id1", "vector": [...], "metadata": {...}}
```

### Update
```python
collection.update(id="id1", vector=[...], metadata={"category": "updated"})
```
- Some databases require full reindex after update
- Others support partial metadata updates without re-embedding

### Delete
```python
collection.delete(ids=["id1", "id2"], filter={"category": {"$eq": "obsolete"}})
```
- Often implemented as tombstone (logical delete) + periodic compaction
- Immediate visibility of deletion depends on consistency model

## Metadata Filtering

Vector search alone is rarely enough — you need to filter results by structured fields.

### Filter Types
| Filter | Example | Performance |
|--------|---------|-------------|
| Equality | `category == "docs"` | Fast with index |
| Range | `date >= "2024-01-01"` | Moderate |
| Set membership | `status IN ["active", "pending"]` | Fast with bitmap index |
| Full-text | `title CONTAINS "machine learning"` | Requires inverted index |
| Geo-spatial | `distance(lat, lon, query) < 10km` | Requires geo-index |
| Nested/JSON | `metadata.author.name == "Smith"` | Slower, DB-dependent |

### Filter Execution Strategies
- **Pre-filter**: apply filter first, then search within results. Requires metadata index. Best when filter is highly selective (<5% of data survives).
- **Post-filter**: run ANN search (2-5× K), then filter. No metadata index needed. Risk: filtered results may not fill top-K.
- **Hybrid**: parallel vector + filter index, merge. Best of both but most complex.

## Key Metrics

### Recall@K
Fraction of true nearest neighbors found by ANN search vs exact.
```
recall = |ANN_results ∩ Exact_results| / K
```
Target: 0.90-0.99 depending on use case.
Test on real data, not synthetic — distribution matters.

### Query Latency
| Percentile | Meaning | Target |
|------------|---------|--------|
| P50 | Median user experience | < 20ms |
| P95 | Slow tail | < 50ms |
| P99 | Worst case | < 100ms |
| P999 | Outliers | < 500ms |

### Throughput (QPS)
Queries per second the system handles at target latency.
Scales with: node count, replica count, index efficiency.

### Index Build Time
Time to build index from scratch.
- Flat: 0 (no index)
- HNSW: O(N log N), minutes for 10M vectors
- IVF: O(N × k × iterations), fast
- DiskANN: O(N log N), slow (disk-bound)

## Data Lifecycle

```
Ingest → Embed → Index → Query → Update/Delete → Re-index → Archive
```

1. **Ingest**: raw documents/chunks arrive (batch or streaming)
2. **Embed**: embedding model converts to vector
3. **Index**: vectors added to index (inline or batch rebuild)
4. **Query**: ANN search with optional metadata filters
5. **Update/Delete**: individual vector mutations
6. **Re-index**: periodic full rebuild for optimal index quality
7. **Archive**: cold-storage for old/rarely-queried vectors

## Key Design Decisions

1. **Dimension**: higher = more expressive but slower, more memory. Match embedding model output.
2. **Distance metric**: must match training loss. Cosine is default for text.
3. **Index type**: HNSW for <10M, IVF for 10M-100M, DiskANN for 1B+.
4. **Index parameters**: tune for recall-latency trade-off on YOUR data.
5. **Filter strategy**: pre-filter for selective filters, post-filter for cheap filters.
6. **Batch vs streaming**: batch for bulk load, streaming for real-time ingestion.
7. **Consistency**: eventual for search (default), strong for critical updates.
8. **Scale strategy**: vertical first, then horizontal with sharding.

## Common Mistakes

- Using cosine when model was trained with dot product (or vice versa)
- Not normalizing vectors when required by distance metric
- Building index once and never re-evaluating parameters as data grows
- Ignoring metadata filter selectivity in index and query design
- Treating vector DB as a black box without understanding index internals
