# PRD Collaboration

## Stakeholder Involvement

### Roles and Responsibilities
| Role | Responsibility | Approval |
|------|----------------|----------|
| Product Manager | Authors PRD, owns vision | Yes |
| Engineering Lead | Feasibility, effort estimation | Technical sign-off |
| Design Lead | UX/UI approach | Design sign-off |
| QA Lead | Test strategy input | Testability input |
| Business Owner | Budget, strategic alignment | Final approval |

### Engagement Timeline
```
Brief → PRD Draft → Review Cycle → Finalize → Handoff
  ^        ^            ^             ^
  PM       PM       All teams      All teams
```

## Review Process

### Review Cycles
1. **Internal Review** (PM + Engineering + Design)
   - Validate feasibility and approach
   - Identify gaps and risks
   - Refine scope and priorities

2. **Extended Review** (All stakeholders)
   - Business alignment check
   - Dependency identification
   - Timeline validation

3. **Final Review** (Leadership)
   - Resource commitment
   - Budget approval
   - Go/No-go decision

### Review Checklist
- [ ] Problem statement is clear and scoped
- [ ] Success criteria are measurable
- [ ] User stories cover happy path and edge cases
- [ ] Technical feasibility confirmed
- [ ] Dependencies identified and planned
- [ ] Open questions documented with owners

## Version Control

### PRD Evolution
- PRDs are living documents — update as understanding grows
- Track changes with version history
- Major changes require re-approval
- Archive superseded versions

### Change Management
| Change Type | Process |
|-------------|---------|
| Minor clarification | Inline update, version bump |
| Scope addition | Re-review with stakeholders |
| Scope reduction | Document rationale, notify team |
| Timeline change | Update timeline section, notify all |
| Cancellation | Archive document, notify all |

## Handoff

### To Engineering
- Final PRD in shared location (Docs, Notion, Confluence)
- Kickoff meeting with PRD walkthrough
- Linked technical spec and design files
- Agreed success criteria and acceptance criteria

### To Design
- User flows and wireframe requirements
- Design system constraints
- Accessibility requirements
- Edge cases and error states
