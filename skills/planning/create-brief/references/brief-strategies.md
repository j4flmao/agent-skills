# Brief Writing Strategies

## Purpose

A product brief defines the problem, opportunity, and high-level approach before detailed specification work begins. It aligns stakeholders on scope before investing in PRDs or technical specs.

## Brief Types

| Type | Length | Audience | Use Case |
|------|--------|----------|----------|
| Problem Brief | 1-2 pages | Product team | Define problem before solution |
| Solution Brief | 2-4 pages | Engineering + Design | Outline approach and scope |
| Opportunity Brief | 1-3 pages | Leadership | Frame investment decision |
| Exploratory Brief | 2-5 pages | Research team | Define exploration scope |

## Structure Templates

### Problem Brief
```
1. Problem Statement
2. User Impact (who, how many, how severe)
3. Business Impact (revenue, retention, NPS)
4. Success Criteria (measurable outcomes)
5. Constraints (time, resources, technical)
6. Stakeholders (who needs to be involved)
```

### Solution Brief
```
1. Problem Summary (1 paragraph)
2. Proposed Solution (2-3 sentence overview)
3. Key User Flows (happy path + edge cases)
4. Technical Approach (architecture sketch)
5. Dependencies (internal systems, third parties)
6. Risks and Mitigations
7. Timeline Estimates
```

## Success Criteria Patterns

### Leading Indicators
- User engagement (DAU/MAU, session length)
- Feature adoption rate
- Completion rate for key flows
- Time to complete primary action

### Lagging Indicators
- Revenue impact
- Retention/Churn rate
- Customer satisfaction (NPS, CSAT)
- Support ticket volume

## Common Mistakes

- Writing solution before defining the problem
- Including too much technical detail
- Lack of measurable success criteria
- Not identifying stakeholders early
- Skipping constraints documentation
- Over-scoping the initial brief
- No clear decision criteria
