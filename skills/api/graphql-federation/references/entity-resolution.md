# Entity Resolution

## Overview
Entity resolution is the mechanism by which a federated gateway assembles a single entity from fields spread across multiple subgraphs. When a client requests fields from an entity that originates in one subgraph and extends in others, the router calls `_entities` on each relevant subgraph, passing entity representations (`__typename` + key fields), and merges the responses.

## Resolution Flow

### Step by Step
```
Client Query: { user(id: "1") { name orders { total } } }

1. Router identifies User entity spans Accounts and Orders subgraphs
2. Router sends to Orders subgraph:
   _entities(representations: [{__typename: "User", id: "1"}])
     { ... on User { orders { total } } }
3. Orders subgraph calls __resolveReference with {id: "1"}
4. Orders subgraph returns: {orders: [{total: 29.99}, {total: 49.99}]}
5. Router merges: Accounts returned {name: "Alice"}, Orders returned {orders: [...]}
6. Router assembles final response: {user: {name: "Alice", orders: [...]}}
```

### The _entities Query
The router generates this automatically. A subgraph never explicitly defines `_entities` — Apollo Server handles it when type definitions include `@key`.

```graphql
# Generated by Apollo Server — do not define manually
extend type Query {
  _entities(representations: [_Any!]!): [_Entity]!
}

scalar _Any
```

## Origin Subgraph Reference Resolver

### Full Entity Resolution
The origin subgraph resolves the full entity state:

```typescript
const resolvers = {
  User: {
    __resolveReference(ref, context) {
      return context.dataSources.users.findById(ref.id);
    },
  },
};
```

```python
# Python (Ariadne / Strawberry)
@user_reference_resolver
def resolve_user_reference(ref: dict, info: Info):
    return db.users.find_by_id(ref["id"])
```

### What __resolveReference Receives
```typescript
// The ref parameter contains the key fields defined in @key
// For @key(fields: "id"): ref = {__typename: "User", id: "1"}
// For @key(fields: "sku"): ref = {__typename: "Product", sku: "SKU-001"}
// For @key(fields: "id userId"): ref = {__typename: "Order", id: "1", userId: "42"}
```

### Caching Reference Resolutions
```typescript
const resolvers = {
  User: {
    __resolveReference: memoize(
      (ref, context) => context.dataSources.users.findById(ref.id),
      { ttl: 5000 } // 5 second cache
    ),
  },
};
```

## Extension Subgraph Reference Resolver

### Minimal Passthrough
Extension subgraphs that add fields but don't need to look up the entity:

```typescript
const resolvers = {
  User: {
    __resolveReference(ref) {
      // Just pass through the reference — field resolvers handle the rest
      return { id: ref.id };
    },
    orders(parent, _, context) {
      return context.dataSources.orders.findByUserId(parent.id);
    },
  },
};
```

### Extension with Own Data Lookup
When the extension subgraph needs its own data:

```typescript
const resolvers = {
  User: {
    // Must still return the entity so field resolvers can access it
    __resolveReference(ref, context) {
      const user = context.dataSources.analytics.getUserProfile(ref.id);
      return { id: ref.id, ...user };
    },
    engagementScore(parent) {
      // parent includes data from the resolved reference
      return parent.engagementScore;
    },
  },
};
```

## @requires Resolution Pattern

### Declaration
```graphql
# Shipping subgraph: needs weight (from Catalog) to compute shippingCost
extend type Product @key(fields: "id") {
  id: ID! @external
  weight: Int @external
  shippingCost: Float @requires(fields: "weight")
}
```

### Router Behavior
1. Client requests `shippingCost` on Product
2. Router queries Catalog subgraph for `weight`
3. Router creates representation: `{"__typename": "Product", "id": "1", "weight": 10}`
4. Router sends to Shipping subgraph's `_entities`
5. Shipping resolver computes using `ref.weight`

### Resolver Implementation
```typescript
const resolvers = {
  Product: {
    __resolveReference(ref) {
      // ref includes weight (pre-fetched by router)
      return { id: ref.id, weight: ref.weight };
    },
    shippingCost(parent) {
      // Compute shipping cost using the pre-fetched weight
      const baseRate = 5.00;
      const perKg = 1.50;
      return baseRate + (parent.weight * perKg);
    },
  },
};
```

## @provides Resolution Pattern

### Declaration
```graphql
# Products subgraph: topProducts includes price and currency inline
# No need for extra fetch to get price
extend type Product @key(fields: "id") {
  id: ID! @external
  name: String! @external @provides(fields: "name")
  price: Float! @provides(fields: "currency")
}

type Query {
  topProducts: [Product!]!
}
```

### How @provides Reduces Fetch Hops
```
Without @provides:               With @provides:
Client                           Client
  │                                │
  ├─ Products (id only)            ├─ Products (id + name + price)
  │   └─ Catalog (name,price)      │   └─ (no extra fetch)
  └─ Reviews                       └─ Reviews
```

### Resolver Implementation
```typescript
const resolvers = {
  Product: {
    __resolveReference(ref) {
      // For topProducts, ref comes from Products subgraph with provides fields
      return { id: ref.id, name: ref.name, price: ref.price };
    },
  },
};
```

## Multi-Key Entity Resolution

### Declaration
```graphql
type Product @key(fields: "id") @key(fields: "sku") @key(fields: "upc") {
  id: ID!
  sku: String!
  upc: String!
  name: String!
}
```

### Resolver Handling All Keys
```typescript
const resolvers = {
  Product: {
    __resolveReference(ref, context) {
      if (ref.id) return context.dataSources.products.findById(ref.id);
      if (ref.sku) return context.dataSources.products.findBySku(ref.sku);
      if (ref.upc) return context.dataSources.products.findByUpc(ref.upc);
      return null;
    },
  },
};
```

The router may pass any of the defined key sets depending on which subgraph the request originates from.

## Non-Resolvable Entity References

```graphql
# Inventory subgraph only references Product — never resolves it
type Product @key(fields: "id", resolvable: false) {
  id: ID! @external
}
```

The router never sends `_entities` queries to this subgraph for Product. This is useful when a subgraph only needs to reference an entity ID for joins.

## @interfaceObject Resolution

```graphql
# Media subgraph defines interface and implementations
interface Media {
  id: ID!
  title: String!
}
type Book implements Media @key(fields: "id") {
  id: ID!; title: String!; author: String!
}
type Movie implements Media @key(fields: "id") {
  id: ID!; title: String!; director: String!
}

# Reviews subgraph adds averageRating to ALL Media types
# Uses @interfaceObject to treat the interface like an entity
type Media @interfaceObject @key(fields: "id") {
  id: ID!
  averageRating: Float!
}
```

```typescript
const resolvers = {
  Media: {
    __resolveReference(ref, context) {
      return { id: ref.id };
    },
    averageRating(parent, _, context) {
      return context.dataSources.reviews.getAverageRating(parent.id);
    },
  },
};
```

## Batch Entity Resolution with DataLoader

### Problem: N+1 Entity Resolution
Without batching, resolving 100 reviews each with a Product entity causes 100 individual resolution calls:

```typescript
// BAD — one DB query per entity
const resolvers = {
  Product: {
    __resolveReference(ref, context) {
      return db.products.findById(ref.id);  // Called 100 times for 100 reviews
    },
  },
};
```

### Solution: DataLoader Batching
```typescript
import DataLoader from 'dataloader';

class ProductLoader {
  private loader = new DataLoader(async (ids: readonly string[]) => {
    const products = await db.products.findByIds([...ids]);
    const map = new Map(products.map(p => [p.id, p]));
    return ids.map(id => map.get(id) ?? null);
  });

  resolve(ref: { id: string }) {
    return this.loader.load(ref.id);
  }
}

const resolvers = {
  Product: {
    __resolveReference(ref, _, { loaders }) {
      return loaders.products.resolve(ref);
    },
  },
};
```

### Batch Context Pattern
```typescript
// Create loaders per request in context
const server = new ApolloServer({
  typeDefs,
  resolvers,
  context: () => ({
    loaders: {
      users: new DataLoader(ids => batchFind('users', ids)),
      products: new DataLoader(ids => batchFind('products', ids)),
      orders: new DataLoader(ids => batchFind('orders', ids)),
    },
  }),
});

const resolvers = {
  User: {
    __resolveReference(ref, _, { loaders }) {
      return loaders.users.load(ref.id);
    },
  },
  Product: {
    __resolveReference(ref, _, { loaders }) {
      return loaders.products.load(ref.id);
    },
  },
};
```

## Entity Resolution Security

### Authorization in __resolveReference
Always check permissions in entity resolution — never assume the router has already authenticated:

```typescript
const resolvers = {
  User: {
    __resolveReference(ref, context) {
      // Validate the reference itself
      if (!ref.id || typeof ref.id !== 'string') return null;

      // Authorization check: can the requesting user access this entity?
      if (!context.auth.canViewUser(ref.id)) return null;

      // Rate limit entity resolution
      if (context.auth.entityQuotaExceeded()) {
        throw new GraphQLError('Entity resolution limit exceeded', {
          extensions: { code: 'RATE_LIMITED' },
        });
      }

      return context.dataSources.users.findById(ref.id);
    },
    email(parent, _, context) {
      // Additional field-level permission check
      if (parent.id !== context.auth.userId && !context.auth.isAdmin()) return null;
      return parent.email;
    },
  },
};
```

### Entity List Bombing Protection
```yaml
# router.yaml
demand_control:
  strategy: cost_bound
  scoring:
    max_cost: 1000

# Apollo Router entity limits
preview_entity_cache:
  enabled: true
  max_entities: 100
```

## Troubleshooting Entity Resolution

| Symptom | Likely Cause | Fix |
|---------|-------------|-----|
| Null for extended fields | __resolveReference returns null | Check DB connectivity, key format |
| Wrong data in merged result | Key mismatch across subgraphs | Verify @key field sets match exactly |
| Slow queries with many entities | N+1 resolution (no DataLoader) | Implement batch loading |
| Entity resolution timeout | Subgraph too slow | Add caching, optimize DB query |
| _entities query fails | Subgraph doesn't have @key | Add @key to the type definition |
| Circular resolution | Both subgraphs extend each other | Break cycle with query-level access |

## Key Points
- __resolveReference is the core mechanism for entity resolution in federation
- Origin subgraphs return full entity; extension subgraphs return just enough for their fields
- @requires causes the router to pre-fetch fields and include them in the representation
- @provides reduces fetch boundaries by embedding field values inline
- Multi-key entities (@key declared multiple times) require handling all key variants
- Non-resolvable keys (resolvable: false) prevent unnecessary _entities queries
- DataLoader batching prevents N+1 during entity resolution
- Authorize in __resolveReference — never assume the router has already checked
- @interfaceObject allows adding fields to all implementations of an interface
- Entity list bombing protection prevents excessive entity resolution calls
